Chấm bài thuật toán — vượt qua kiểm tra I/O

Tài liệu cho giảng viên và đội kỹ thuật: các cách chấm bài lập trình thuật toán, từ so khớp đầu vào/đầu ra đến đánh giá sinh viên có giải đúng bài toán trong đề hay không.

1. Tổng quan

Hệ thống chấm trực tuyến (Online Judge, OJ) phổ biến nhất dựa trên black-box testing: chạy chương trình sinh viên với tập test, so stdout với đáp án mong đợi. Cách này công bằng, có thể mở rộng, nhưng không đủ khi mục tiêu giáo dục là:

  • Sinh viên chọn đúng ý tưởng thuật toán (DP, đồ thị, tham lam…).
  • Độ phức tạp đáp ứng ràng buộc đề (ví dụ O(n log n), không brute O(2n).
  • Code phản ánh đúng bài toán (tổng, modulo, đếm đường đi…) chứ không “ăn may” trùng output trên vài test.
  • Tránh gian lận: hardcode, copy, giải sai bài nhưng lọt public test.
Kết luận then chốt: Không có phương pháp tự động nào chứng minh 100% rằng sinh viên “hiểu đề”. Cần kết hợp nhiều lớp: test mạnh, checker/oracle, giới hạn tài nguyên, rubric thủ công, và (khi cần) phỏng vấn.

Đối tượng đọc:

  • Giảng viên — chọn chiến lược chấm, thiết kế đề, rubric, tỷ lệ điểm auto/thủ công.
  • Người xây hệ thống — triển khai trên code-judge và lộ trình mở rộng.

Lớp 1 — Đúng/sai hành vi (behavioral correctness)

Output khớp trên từng test, không TLE/MLE/RE, đúng định dạng I/O. Đây là nền tảng của mọi OJ; trong code-judge worker so sánh sau chuẩn hóa output.

Chứng minh đúng đề? Chỉ khi tập test đủ mạnh và đề không mơ hồ; thường chỉ chứng minh “đúng trên tập đã cho”.

Lớp 2 — Đúng bản chất thuật toán (semantic / relational correctness)

Không chỉ so chuỗi output mà xác minh quan hệ input → output theo đặc tả: checker tùy chỉnh, lời giải mẫu (oracle), property-based test, stress với brute force trên n nhỏ.

Chứng minh đúng đề? Mạnh hơn lớp 1; vẫn có thể bị hardcode nếu test quá ít hoặc checker yếu.

Lớp 3 — Đúng yêu cầu mô tả / đúng đề (requirement alignment)

Đánh giá sinh viên có làm đúng những gì đề yêu cầu: thuật toán bắt buộc, giải thích độ phức tạp, cấu trúc code, không hack. Cần rubric, traceability (FR-*), AST policy (Tree-sitter) cho rule cấm/bắt buộc, chấm thủ công hoặc AI có người giám sát.

Chứng minh đúng đề? Gần mục tiêu giáo dục nhất; tốn công, khó scale 100% tự động.

3.1. Black-box I/O (nền tảng OJ)

Mỗi test case gồm input, expectedOutput, weight, isHidden. Worker chạy từng test (Lambda/Judge0), chuẩn hóa rồi so sánh: 

function normalizeOutput(out: string): string {
  if (!out) return '';
  return out
    .replace(/\r\n/g, '\n')
    .split('\n')
    .map((line) => line.trimEnd())
    .join('\n')
    .trim();
}

Nguồn: apps/worker/src/index.ts — normalizeOutput.

Khi nào chứng minh đúng đề? Khi bộ test cover đủ không gian input (biên, typical, adversarial) và đề định nghĩa output duy nhất. Khi nào chỉ đúng output? Hầu hết trường hợp thực tế nếu hidden yếu.

3.2. Checker / validator tùy chỉnh

Thay vì actual === expected, chạy chương trình checker nhận (input, output_cho_lam, output_hs) và trả ACCEPT / WRONG. Phổ biến trong ICPC (testlib, custom checker).

  • So số thực: |a − b| ≤ ε.
  • Nhiều đáp án hợp lệ: so multiset, hoán vị, hoặc “tồn tại labeling”.
  • Output lớn: checker đọc cấu trúc thay vì diff từng byte.

Đúng đề: Checker mã hóa đúng định nghĩa toán trong đề. Chỉ đúng output: Checker chỉ so format superficial.

3.3. Golden solution (oracle)

Lời giải mẫu tin cậy chạy trên cùng input để tạo hoặc xác minh expectedOutput. Trong code-judge: golden-verify-job.ts + AI diagnose khi verify fail (golden-verify-failure-diagnose.prompt.ts — golden được coi đúng cho WRONG_ANSWER, gợi ý sửa expected).

Đúng đề: Golden đúng và test đại diện đủ lớp input. Chỉ đúng output: HS có thể dùng thuật toán khác vẫn đúng — oracle không “chấm ý tưởng”, chỉ chấm hành vi.

3.4. Property-based & stress testing

Sinh hàng nghìn input ngẫu nhiên (có seed cố định), kiểm tra:

  • sort(sort(x)) === sort(x)
  • Tổng trước/sau thao tác không đổi
  • Kết quả HS === brute khi n ≤ Nmax

Phát hiện lỗi logic mà vài test tay không thấy. Đúng đề nếu property được trích từ đề bài chính xác.

3.5. Giới hạn thời gian / bộ nhớ

Ràng buộc timeLimitMs, memoryLimitMb trên Problem loại bỏ brute không chịu được input lớn. Không đảm bảo HS dùng đúng thuật toán mong muốn (có thể O(n²) lọt nếu limit rộng).

Calibrate từ golden (code-judge)

  • Admin: POST /problems/:id/calibrate-limits — worker chạy golden trên mọi testcase qua Judge0, trả suggestedTimeLimitMs / suggestedMemoryLimitMb (hệ số an toàn LIMIT_TIME_SAFETY, LIMIT_MEM_SAFETY).
  • Worker áp dụng hệ số ngôn ngữ khi chấm (Python/Java ×2 mặc định) so với limit trên đề.
  • Contest: ContestProblem.timeLimitMsOverride / memoryLimitMbOverride được worker đọc khi submission.contestId có.
  • Verdict: TLE (Judge0 status 5), MLE (mô tả/message memory) → TimeLimitExceeded / MemoryLimitExceeded.

Memory và cgroup (Judge0)

Ubuntu 24 + JUDGE0_USE_CGROUP=false + isolate_stub: time enforce được; memory chủ yếu hiển thị trên đề. Muốn MLE thật: chuyển host sang cgroup v1 + isolate thật — xem JUDGE0-CGROUP-V1-MIGRATION.md.

3.6. Static analysis & anti-hardcode (regex / heuristic)

Heuristic trên chuỗi source (không parse cú pháp):

  • Chuỗi if (n == 5) lặp với nhiều hằng — nghi hardcode theo test.
  • Mảng ánh xạ kích thước = số test public.
  • Tìm tên hàm cấm bằng regex — dễ nhầm comment/string.

Phù hợp prototype nhanh. Với rule cấm API hoặc bắt buộc cấu trúc, nên dùng mục 3.8 AST + Tree-sitter để giảm false positive.

3.8. AST, Tree-sitter và policy engine

Hướng nâng cấp static analysis: parse mã nguồn thành cây cú pháp bằng Tree-sitter, sau đó chạy policy rules (query hoặc visitor) để chặn phần không được phép và (tuỳ chọn) rút dấu hiệu cấu trúc thuật toán — không thay black-box I/O. 

flowchart LR
  src[Source_code] --> parse[Tree_sitter_parse]
  parse --> tree[CST_syntax_tree]
  tree --> policy[Policy_engine]
  policy -->|vi_pham| pv[POLICY_VIOLATION]
  policy -->|pass| fp[Structural_fingerprint]
  fp --> io[Blackbox_IO_judge]
  io --> score[Diem_tong]

Vai trò trong ba lớp đánh giá

  • Lớp 3 (chính): enforce “đúng khung bài” — cấm sort, bắt đệ quy, cấm numpy…
  • Lớp 2 (phụ): fingerprint cấu trúc (có vòng lồng, có đệ quy) — gợi ý DP/graph cho GV, không thay test.
  • Lớp 1: Không thay thế so output; policy thường chạy trước Lambda để fail nhanh, tiết kiệm tài nguyên.
Lưu ý: AST/CST chứng minh cú pháp và policy, không chứng minh sinh viên “hiểu đề” hay logic toán đúng. Hai bài khác thuật toán có thể cùng fingerprint; vẫn bắt buộc hidden test + rubric khi cần.

; Cấm gọi sorted() — chỉ match call thật, không match text trong comment

  function: (identifier) @fn
  (#eq? @fn "sorted"))

Cấu trúc thuật toán: không dùng AST thô

AST/CST gốc quá chi tiết (tên biến a vs b, vị trí dòng). Để so “giống DP / giống golden”, chuẩn hóa thành structural fingerprint:

  • Đổi identifier → VAR, literal → NUM
  • Vector đặc trưng: [has_recursion, max_loop_depth, forbidden_call_count, …]
  • So với template từ lời giải mẫu hoặc ngưỡng từng đề (policy pack)

Khi nào chứng minh đúng đề? Khi policy pack được viết bám FR-* trong đề (ví dụ FR-3: cấm sort) và GV chấp nhận “đúng khung” là đủ cho phần điểm rubric. Khi nào chỉ đúng khung? Hầu hết trường hợp — logic vẫn cần I/O test.

Kết quả chấm trên OJ

Nên tách trạng thái POLICY_VIOLATION (hoặc RuleViolation) khỏi Wrong Answer, kèm danh sách rule id đã vi phạm để sinh viên sửa đúng hướng (“Bài cấm dùng sort có sẵn”). Có thể: 0 điểm auto khi vi phạm bắt buộc, hoặc vẫn chạy I/O nhưng trừ rubric.

Rủi ro và cách giảm

  • False positive: match đúng loại node (call vs identifier trong biểu thức).
  • Đa ngôn ngữ: mỗi languageId một grammar + file query .scm.
  • Obfuscation: code cố tình rối; policy mạnh nhất ở rule cấm rõ, không “đoán ý đồ”.
  • C/C++ macro: có thể cần parse sau preprocessor hoặc chỉ hỗ trợ subset không macro.
  • Bảo trì: Problem.policyPackId + version, đổi policy không làm hỏng bài cũ.

3.7. AI sinh test & review

Pipeline ai-testcase.prompt.ts: LLM trả JSON test case bám statement, IO format, biên. Bắt buộc: golden verify trước khi publish; không tin LLM một mình.

AI review code vs đề: dùng rubric + statement; không đưa expected hidden vào prompt (tham chiếu AI-HINT-ASSISTANT). Kết quả là gợi ý cho GV, không chấm điểm cuối tự động.

4. Bán tự động và chấm thủ công (nội dung / đúng đề)

4.1. Rubric có trọng số

Ví dụ rubric môn Thuật toán:

Tiêu chíTrọng sốMô tả
Correctness (auto)50–70%Test I/O + hidden
Algorithm choice15–25%Đúng họ thuật toán (DP, graph…)
Complexity analysis10–15%Chứng minh O(·), không chỉ “chạy nhanh”
Code quality5–10%Đặt tên, tách hàm, edge case
Documentation0–10%Comment invariant, báo cáo ngắn

Công thức minh họa: Điểm = 0.65 × Điểm_auto + 0.35 × Điểm_rubric.

4.2. Requirement traceability (FR-*)

Mô hình PROJECT testcase trong repo: mỗi yêu cầu FR-1, FR-2… map tới test trong testManifest (ai-project-testcase.prompt.ts). Áp dụng tư duy cho bài thuật toán:

  • FR-1: “Giải bằng quy hoạch động một chiều”
  • FR-2: “Độ phức tạp O(n log n)” — kiểm tra bằng TLE trên n = 106
  • FR-3: “Không dùng thư viện sort có sẵn” — policy AST (Tree-sitter) + TLE

Giảng viên chấm: test auto cover FR nào; FR còn lại chấm tay hoặc oral.

4.3. Peer review & bài nộp kèm meta

  • Sinh viên nộp kèm: pseudocode, giải thích độ phức tạp, 1–2 test tự viết.
  • Peer review theo checklist (đọc được, logic khớp code).
  • Milestone: draft (feedback) → final (chấm điểm).

4.4. Phỏng vấn / oral exam

Khi điểm auto cao nhưng nghi ngờ gian lận, hoặc bài “thiết kế thuật toán”: HS giải thích 5 phút ý tưởng, invariant, độ phức tạp. Đây là cách mạnh nhất cho lớp 3.

5. Phát hiện “đúng test nhưng sai đề”

Các pattern phổ biến trong giảng dạy và contest:

PatternTriệu chứngPhòng ngừa
Hardcode theo test if (n==10) print("42"); bảng map input→output Hidden đa dạng; static scan; checker
Brute chỉ đủ test nhỏ AC public, TLE/WA hidden lớn Subtask; time limit chặt; stress
Sai bài toán, trùng output Tính tổng thay vì tổng modulo; đếm sai nhưng trùng vài case Property test; nhiều seed; checker logic
Copy / không hiểu Code giống MOSS; không giải thích oral Plagiarism + oral; đổi biến đề

Lách policy AST Vòng lặp rỗng, gọi hàm wrapper tên khác Rule forbid_call theo module; vẫn cần I/O mạnh

Gợi ý khi ra đề (giảng viên)

  1. Viết đặc tả output không mơ hồ (modulo, thứ tự, format dòng).
  2. Tối thiểu: public (hướng dẫn) + hidden (chấm điểm) + biên (0, 1, max n).
  3. Ưu tiên checker khi nhiều đáp án đúng.
  4. Nếu đề cấm API/cấu trúc: viết policy pack (Tree-sitter), không chỉ regex.
  5. Yêu cầu phân tích độ phức tạp trong báo cáo (lớp 3).
  6. Golden verify mọi test trước khi mở bài cho sinh viên.
  7. Policy AST: thử policy trên golden và vài mã “cố ý vi phạm” trước khi publish.

7. Ánh xạ triển khai trên code-judge

Phần này mô tả hiện trạng trong repo và gợi ý mở rộng — không thay đổi code trong tài liệu này.

Luồng chấm submission (hiện tại và đề xuất có AST)

Hiện tại — chỉ I/O sau enqueue:

sequenceDiagram
  participant Web
  participant API as core_api
  participant Queue as BullMQ
  participant Worker
  participant Judge as Lambda_Judge0
  Web->>API: POST submission
  API->>Queue: enqueue
  Queue->>Worker: processSubmission
  loop each TestCase
    Worker->>Judge: run source + input
    Judge-->>Worker: stdout stderr status
    Worker->>Worker: normalizeOutput compare
  end
  Worker->>API: score status per case

Đề xuất — policy trước Lambda:

sequenceDiagram
  participant Worker
  participant Policy as policy_service_TreeSitter
  participant Judge as Lambda_Judge0
  Worker->>Policy: parse plus run policyPack
  alt policy fail
    Policy-->>Worker: POLICY_VIOLATION plus ruleIds
  else policy pass
    Policy-->>Worker: ok plus optional fingerprint
    loop each TestCase
      Worker->>Judge: run source plus input
      Judge-->>Worker: stdout status
    end
  end

Gợi ý schema / API (roadmap)

// Problem (mở rộng)
policyPackId?: string;      // ví dụ "no-sort-v1", "require-recursion-v1"
policyMode?: "block" | "warn"; // block = không chạy test

// Kết quả submission
policyResults?: {
  passed: boolean;
  violations: { ruleId: string; message: string; line?: number }[];
  fingerprint?: string;   // optional, cho GV
};

  • apps/core-api/src/ai-testcase/ai-project-testcase.prompt.ts — FR-* + testManifest (PROJECT)

8. Phụ lục

8.1. Checklist giảng viên (thiết kế bài + test)

  • Đặc tả input/output rõ ràng, có ví dụ minh họa
  • Giới hạn n, time, memory khớp thuật toán mong đợi
  • Public test hướng dẫn; hidden đa dạng (biên, max, adversarial)
  • Golden solution + verify toàn bộ test trước publish
  • Cân nhắc checker nếu nhiều đáp án đúng
  • Rubric / FR-* nếu bài yêu cầu ý tưởng hoặc cấm thư viện
  • Policy pack Tree-sitter nếu đề cấm API / bắt buộc cấu trúc (xem §3.8)
  • Kế hoạch oral hoặc plagiarism nếu điểm cao có giá trị

8.2. Checklist kỹ thuật (vận hành OJ)

  • Worker normalizeOutput thống nhất với đề (newline, trim)
  • Hidden test không lộ qua API/UI sinh viên
  • Lambda timeout > time limit bài + buffer
  • Golden verify job chạy khi đổi test hoặc golden
  • AI testcase: không publish khi verify fail
  • Log per-case cho GV (không chỉ Accepted/Wrong)
  • Backup test case và golden trong DB/MinIO
  • Policy pack versioned; grammar Tree-sitter khớp languageId submission

8.3. Tài liệu tham khảo

  • Tree-sitter — parser đa ngôn ngữ, query .scm
  • ICPC — testlib / custom checker pattern
  • QuickCheck / Hypothesis — property-based testing
  • Stanford MOSS — similarity detection (có thể kết hợp AST hash)
  • Codeforces / AtCoder — partial scoring, subtasks
  • Trong repo: PROJECT-TESTCASE-IMPLEMENTATION-OVERVIEW
