Kế hoạch sinh testcase cho bài PROJECT (Backend / Frontend)

Dự án: code-judge Phiên bản tài liệu: 1.0 Ngày: 2026-05-18 Kế hoạch — chưa triển khai

Blueprint sản phẩm và kỹ thuật cho pipeline giảng viên tạo gói hidden tests gắn với bài ProblemMode.PROJECT, tích hợp hệ auto-grader (MinIO, BullMQ, Docker sandbox) và mở rộng module AI / golden hiện có.

1. Tóm tắt điều hành

Mục tiêu

Cho phép giảng viên hoặc admin tạo và xuất bản một bundle testcase ẩn (hidden_tests.zip) trước khi học viên nộp mã nguồn. Bundle này được worker mount read-only vào sandbox Docker khi chấm bài, tương tự đặc tả auto-grader: working_dir/hidden_tests → /app/tests.

Phạm vi v1

  • Trong phạm vi: Backend Node (Jest/Vitest + supertest), Frontend (Playwright e2e), fullstack đơn giản (một repo zip).
  • Ngoài phạm vi v1: Mobile native, monorepo đa package phức tạp, ngôn ngữ không Node (Java/Python project judge riêng).

So sánh ALGO vs PROJECT

Khía cạnhALGOPROJECT
Đơn vị test TestCase.input / expectedOutput (text) Bundle file tests/**/*.spec.ts + fixtures trên MinIO
Engine chấm Judge0 / Lambda từng case Docker: npm test hoặc npm run test:e2e → parse JSON
Sinh test bằng AI Đã có Stub — chỉ plannedObjectKeys
Worker chấm zip N/A Chưa — JUDGE-PROJ-WK-*
Đã có ALGO AI testcase Draft + generate-and-save + Zod schema
Một phần PROJECT generate-and-save Job metadata, không LLM, không insert test thật
Một phần Golden verify stdin/stdout ALGO; chưa Docker project
Chưa có Project judge worker Extract zip, build, test, score breakdown

2. Kiến trúc trong hệ auto-grader

Hệ thống auto-grader gồm: Producer (NestJS API), Storage (MinIO), Broker (Redis + BullMQ), Consumer (Grading Worker), Sandbox (Docker), Result DB (PostgreSQL). Tài liệu này tập trung vào nhánh chuẩn bị hidden tests; nhánh chấm submission dùng cùng bundle đã publish.

3. Đầu vào và đầu ra sinh testcase

Đầu vào (Giảng viên / Admin)

  • Problem metadata: title, statementMd, mode = PROJECT, stack (backend | frontend | fullstack).
  • AssignmentTestConfig (schema mới — chưa có trong Prisma):
    • dockerImage, installCommand, testCommand
    • resultParser: jest-json | playwright-json
    • framework: express, nest, react, next, …
  • Golden solution: zip hoặc upload qua prefix golden-solutions/{problemId}/.
  • Tài liệu bổ sung: PDF/README — presign resourceKind: ai-input (đã có storage API).
  • Starter template (tùy chọn): skeleton học viên để smoke-test “fail có kiểm soát”.

Đầu ra

  • MinIO: problems/{problemId}/hidden-tests/{version}/bundle.zip
  • DB: Problem.hiddenTestsObjectKey, Problem.hiddenTestsVersion, AiGenerationJob.structuredOutput
  • TestCase rows (meta): có thể chỉ lưu tên test, weight, isHidden — không nhét file vào cột text

4. Quy trình sinh testcase (6 bước)

Bước Tên Hành động Thành phần code-judge Trạng thái
1 Thu thập ngữ cảnh Upload spec, chọn stack, bind ai-input StorageController, mở rộng đọc MinIO → prompt Một phần
2 Sinh nháp test files LLM → cây file tests/ + scripts Mở rộng ai-testcase hoặc project-testcase Chưa
3 Hợp nhất golden Đối chiếu API contract với golden zip golden-solutions/ prefix Một phần
4 Xác thực sandbox Queue chạy Docker giống worker chấm Queue golden-verify-project (mới) Chưa
5 Duyệt & publish GV sửa file, approve version Wizard FE (mở rộng AiTestCaseDraftSheet) Chưa
6 Gắn problem Cập nhật object key + version ProblemsService Chưa
Điều kiện publish: (1) Golden pass 100% trong sandbox; (2) Starter template fail có kiểm soát nếu có; (3) Zip vượt qua kiểm tra bảo mật (path traversal, zip bomb — tham chiếu JUDGE-PROJ-TEST-04).

Prompt AI cho PROJECT (khác ALGO)

ALGO dùng schema { testCases: [{ input, expectedOutput, ... }] } trong ai-testcase.prompt.ts. PROJECT cần schema file-based: 

{
  "files": [
    { "path": "tests/api/auth.spec.ts", "content": "..." }
  ],
  "testManifest": [
    { "testName": "Login API", "suite": "api", "weight": 5, "isHidden": true }
  ],
  "notes": "optional assumptions"
}
Ràng buộc prompt (system)
  • Chỉ trả về một JSON object hợp lệ, không markdown.
  • Test không gọi mạng ngoài; không đọc file ngoài /app.
  • Tên test ổn định (describe / it) để parser map details[].
  • Backend: Jest hoặc Vitest + supertest; Frontend: Playwright tối thiểu.
  • Hỗ trợ revision khi parse fail (pattern đã có ở ALGO).
Ví dụ user message (rút gọn) 
<problem_context>
Title: REST API Quản lý thư viện
Stack: backend (NestJS)
Statement: ...
</problem_context>
<assignment_config>
dockerImage: node:18-alpine
testCommand: npm test -- --json --outputFile=result.json
</assignment_config>
<golden_hint>Golden đã implement đủ CRUD Book, Author</golden_hint>

5. Cấu trúc gói hidden_tests.zip

hidden_tests/
  package.json              # devDependencies cho test runner
  jest.config.js            # hoặc vitest.config.ts
  tests/
    api/
      auth.spec.ts
      books.spec.ts
    fixtures/
      seed.sql
  playwright.config.ts      # nếu frontend / e2e
  e2e/
    login.spec.ts
  README_INSTRUCTOR.md      # ghi chú nội bộ (không mount cho SV)

Lệnh chấm theo stack

StackInstallTestParser
Backend npm ci npm test -- --json --outputFile=/app/result.json jest-json
Frontend npm ci npm run test:e2e -- --reporter=json playwright-json

Ánh xạ parser → score

  • score = sum(weight_i * passed_i) / sum(weight_i) * 100
  • Exit code ≠ 0 trước khi có result.json → BUILD_ERROR hoặc CRASHED
  • Timeout 300s → TIMEOUT
Giới hạn bảo mật zip (publish gate)
  • Max uncompressed size (vd. 50MB)
  • Max số file (vd. 500)
  • Từ chối path ../, absolute paths, symlink nguy hiểm

6. API và UI (kế hoạch triển khai)

API đề xuất

MethodPathMô tảAuth

GET /ai-testcase/jobs/:jobId Poll trạng thái + preview file tree Creator / ADMIN

POST /problems/:id/project-testcases/publish Đóng gói zip → MinIO, bump version Creator / ADMIN
GET /problems/:id/project-testcases/download Presigned URL tải bundle Creator / ADMIN

Tái sử dụng codebase

  • buildAiGeneratedTestcaseObjectKeys → mở rộng prefix problems/{id}/hidden-tests/...
  • Queue BullMQ mới project-testcase-gen (tránh block HTTP — xem cảnh báo trong AI doc)
  • Presign ai-input, golden-solutions đã có trong storage module

UI: Wizard 4 bước (PROJECT)

Gắn vào form tạo/sửa problem PROJECT (AdminProblemCreate.tsx, ClassProblemCreate.tsx):

  1. Cấu hình: stack, Docker image, lệnh install/test, parser.
  2. Golden + AI: upload golden zip, nút “Sinh testcase”, poll job.
  3. Preview: cây file, editor đơn giản hoặc download chỉnh sửa local.
  4. Validate & Publish: log sandbox, nút publish khi pass gate.

Mở rộng từ AiTestCaseDraftSheet.tsx (hiện tối ưu cho ALGO input/output).

7. Xử lý lỗi và vận hành

Kịch bảnPhát hiệnHành động pipeline sinh test
LLM output lỗi Zod parse fail Job FAILED; cho phép revision prompt (pattern ALGO)
Golden fail Container exit ≠ 0 Chặn publish; hiển thị stderr/log artifact
Test flake Hai lần validate khác kết quả Cảnh báo GV; gợi ý timeout/selector
Timeout > 300s Kill Docker; job TIMEOUT
Infinite loop / CPU abuse Docker limits Container bị giới hạn; không treo host
Malicious zip Path traversal / bomb Từ chối ở bước upload/publish gate
Queue congestion BullMQ depth cao Scale worker replicas (ops)
Phụ thuộc: Worker chấm submission học viên (JUDGE-PROJ-WK-*) là prerequisite cho điểm số thật trên production, nhưng không chặn pipeline sinh + validate testcase nếu queue validate riêng dùng chung module Docker.
