Giới thiệu & Phạm vi

Tài liệu mô tả bộ kịch bản kiểm thử có cấu trúc cho nền tảng EdTech Online Judge CodeJudge, bao phủ luồng nghiệp vụ, bảo mật, đồng thời, tính toàn vẹn dữ liệu thời gian thực, độ bền tích hợp AI và hành trình người dùng theo vai trò. Mỗi case được thiết kế để phát hiện lỗi logic nghiệp vụ (business logic bugs), không chỉ lỗi UI.

Vùng rủi ro cao (ưu tiên regression mỗi release): IDOR trên Problem/Classroom/Submission, race presign → PUT MinIO → finalize, burst submit cuối contest, mất đồng bộ Socket.io khi chấm, rò rỉ testcase ẩn / đáp án qua AI Hint, và tính penalty Leaderboard khi trạng thái submission thay đổi sau khi chấm xong.

Chú giải cột bảng

Happy Edge False / Negative Security
A S M
  • Loại: Happy = luồng chuẩn; Edge = biên/thiếu tài nguyên; False = hành vi sai phải bị chặn; Security = tấn công/quyền truy cập.
  • Auto: A = Full automation (CI PR); S = Semi-auto (staging, Mailhog, Judge0, k6); M = Manual bắt buộc (OAuth, chaos, đánh giá AI định tính).
  • Layer: Tầng kiểm thử chính — Unit, API, E2E, Load, Chaos, OAuth, …
  • CI: PR = mỗi pull request; Nightly = pipeline đêm; Release = trước production / UAT.
  • Mức độ rủi ro: High = ảnh hưởng điểm thi, dữ liệu người dùng, hoặc sandbox; Medium = trải nghiệm/đồng bộ; Low = hiếm hoặc có workaround.

E. User Journeys (Luồng trọn vẹn theo vai trò)

Các luồng E2E dưới đây liên kết nhiều case trong bảng trên — dùng cho UAT và demo release.

Student — Từ Join lớp đến AC

  1. Onboarding: Google Login / đăng ký email → Dashboard hiển thị lớp (AUTH-H-02, SEC-AUTH-01).
  2. Join lớp: Nhập mã invite hoặc accept email → enrollment ACTIVE (CLS-H-02).
  3. Làm bài ALGO: Mở Classwork → Monaco Editor → Run dry-run (JUD-E-01) → Submit (JUD-H-01).
  4. Realtime: Theo dõi Pending→Running→AC; mô phỏng mất mạng (RT-JUD-01).
  5. Sai bài: WA trên hidden → chỉ thấy [Hidden] (SEC-SUB-02) → AI Hint Socratic (AI-HINT-01, SEC-AI-02 nếu contest).
  6. Contest: Tham gia contest lớp → nộp trong khung giờ → xem leaderboard cập nhật (RT-CON-01, RT-CON-02).
  7. Kết thúc: AC tất cả bài được giao; điểm classwork/contest khớp backend.
Điểm logic cần soi kỹ: Học sinh không được xem đề CONTEST_ONLY khi contest chưa mở hoặc khi chỉ có link problem bank (SEC-PRB-03).

Teacher — Soạn đề AI → Verify → Contest

  1. Tạo lớp: Create Classroom → copy mã / mời email (CLS-H-01).
  2. Soạn đề: AI Generate Problem + AI Testcase (AI-TC-01, AI-TC-02) → chỉnh tay testcase.
  3. Golden Verify: Dán solution mẫu → verify → analyze failure nếu sai (AI-GV-01).
  4. Publish: Gán classwork; học sinh thấy đúng phạm vi PRIVATE (PRB-H-01).
  5. Contest: Tạo contest, set startAt/endAt, max submissions (CON-H-01, CON-SEC-02).
  6. Giám sát: Leaderboard realtime; kiểm tra không leak đề teacher khác (SEC-PRB-01).
  7. Archive: Kết thúc học kỳ → archive lớp → không submit mới (CLS-E-01).

Admin — Quản trị User & System Analytics

  1. Analytics: Dashboard tổng quan — đối chiếu DB (ADM-H-01).
  2. User management: Lock tài khoản vi phạm → user không refresh được (SEC-AUTH-03).
  3. Problem bank: Moderation PUBLIC (PRB-H-03).
  4. Bảo vệ hệ thống: Không tự hạ quyền admin cuối (ADM-F-01).
  5. Vận hành: Redis/MinIO/Judge0 health (INF-01, QUE-CON-01); xem log VPS khi alert Unhealthy.
